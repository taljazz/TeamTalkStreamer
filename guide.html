TeamTalk Streamer — User Guide

TeamTalk Streamer is an accessible Windows application that streams your PC's audio directly into a TeamTalk voice channel. It replaces the traditional "Virtual Audio Cable plus TeamTalk" workaround with a clean, screen-reader-first tool that handles the whole flow — connect, log in, join, stream — in a single click.

This guide walks through setup, daily use, keyboard controls, accessibility features, and troubleshooting. Press F1 at any time from inside the app to reopen this document.

1. Before You Begin

What you need

  • Windows 10 or 11 (64-bit). The .NET 10 runtime is embedded in the executable — no separate install is required.
  • A TeamTalk 5 server to connect to — either self-hosted or one you already have an account on.
  • Your TeamTalk account details — host, nickname, username, password, and the channel you want to stream into.
  • A TeamTalk SDK License Key from BearWare if you plan to use the app beyond the SDK's 30-day trial.

First install

  1. Extract the distribution zip anywhere — your desktop, your documents folder, a USB stick. It doesn't matter where, as long as the three files stay together: TeamTalkStreamer.exe, TeamTalk5.dll, and guide.html.
  2. Run TeamTalkStreamer.exe. You will see the main window — a session-status line at the top, a row of primary buttons, and an advanced sources list.
  3. If a screen reader is running, TeamTalk Streamer will detect it automatically. Tolk supports NVDA, JAWS, Window-Eyes, System Access, ZoomText, Dolphin SuperNova, and the Windows SAPI fallback.
Note on screen-reader output. If the app opens silently and you were expecting spoken announcements, the Tolk native DLLs may not be present alongside the executable. This is optional — the app still works without them — but if you want the spoken state-change feedback, download the Tolk DLLs from the Tolk project and drop them next to TeamTalkStreamer.exe.

2. Configuring Your Server

Before the first stream, tell TeamTalk Streamer which server to use.

  1. Click Server settings…, or press Tab until focus lands on that button and press Enter.
  2. Fill in the Server section:
    • Host — the hostname or IP address of your TeamTalk server.
    • TCP port and UDP port — usually 10333, which is the default for most installs.
    • Server password — leave blank unless the server itself is password-protected.
  3. Fill in the User section — nickname (shown to others in the channel), username, and password. Leave username and password blank for anonymous servers.
  4. Pick a channel using the browser at the bottom of the dialog. Instructions for this are in the next section.
  5. Press Save. Settings are persisted to %APPDATA%\TeamTalkStreamer\settings.json.

Browsing the channel list

The Server settings dialog has a Browse channels section that lets you pick a channel from a live list rather than typing its path. Once you have entered your host and credentials:

  1. Click Probe server for channel list. The app briefly connects to the server, fetches every channel, and disconnects. A status line reports progress; Tolk speaks the final count.
  2. Tab or arrow down into the channel list. Each entry reads as a friendly name — e.g. "Music (under Lobby)" or "General chat — locked" if the channel requires a password.
  3. Select the channel you want. The selection appears in the Selected channel display at the top of the Channel section.
  4. If the channel is password-protected, enter the channel password in the field below the display.

You can still type a path manually into settings.json if you ever need a channel that the browser can't see, but for normal use the browser is the recommended workflow — it avoids path-typo errors and shows which channels require passwords up front.

3. Excluding an App from the Stream

By default TeamTalk Streamer captures your whole system audio mix — whatever your speakers are playing goes into the channel. That includes your screen reader's speech. If you'd rather keep your screen reader (or any other single app) on your side only, use the excluded-apps picker to skip its audio.

What it does

The excluded app's audio is subtracted from the loopback capture before it reaches the TeamTalk channel. You still hear the app normally on your own speakers or headphones — only the stream going out is affected. This is the same mechanism as Windows 11's per-app volume routing but scoped to our capture, so nothing about your day-to-day audio setup changes.

Picking an app to exclude

  1. Start the app you want to exclude (for example, let your screen reader speak something so it's currently producing audio).
  2. Click Excluded apps… on the main window, or press Tab to it and press Enter.
  3. The dialog lists every app currently making sound. If your target isn't there yet, click Refresh — the list only shows sessions that are active at enumeration time.
  4. Select the app and press Save. Tolk confirms the choice aloud.
  5. To remove the exclusion later, reopen the dialog and press Clear.

The header of the dialog always shows the currently-saved exclusion, even if the app isn't running right now. That way you can see at a glance what's configured before picking something new.

Requirements and limits

  • Windows version. The exclusion feature uses Windows' Process Loopback API, introduced in Windows 10 version 2004 (May 2020). Older Windows can't exclude apps; TeamTalk Streamer will tell you if activation fails and fall back to the full-mix capture.
  • One app at a time. The API accepts a single process tree per capture, so you can exclude one app plus all of its child processes — but not two unrelated apps simultaneously. The common case (one screen reader) is covered; multi-app exclusion would need a different architecture.
  • Excluded app not running at stream time? TeamTalk Streamer notices and falls back to the full-mix capture automatically. Tolk announces "[app] is not running; streaming the full system audio mix." so you know the exclusion wasn't applied. The saved exclusion stays configured; it kicks in again the next time the app is running when you start streaming.

4. Streaming

Once settings are configured, streaming is a single button.

  1. Play whatever audio you want to share through your PC's normal speakers / default output — music, a recording, a game, a conference, anything.
  2. Click Start streaming (or press Enter when focus is on it — it is the window's default button).
  3. TeamTalk Streamer runs the full flow in one step: attaches a loopback capture of your default output device, connects to your configured server, logs in, joins the channel, and begins sending the captured PCM into the channel. Every transition is announced by Tolk and marked with a feedback tone.
  4. Click Stop streaming when you are done. The app stops capture, closes the stream, disconnects from the server, and detaches the source.

The Start / Stop button is a toggle — it shows "Start streaming" when idle and "Stop streaming" while connecting, joined, or actively streaming. Pressing it during a transient state (e.g. during connection) will cancel the in-progress flow and return to idle.

Tip. Nothing stops you from playing audio after pressing Start — the loopback captures whatever is playing on your default device at any moment. You can start streaming first, then start your music or game.

6. The Advanced Sources List

Below the primary buttons is a small panel labelled Audio sources (advanced). This shows every source currently attached to the router. Most users never need to interact with it — the Start streaming button attaches and detaches the loopback source automatically.

The panel is there for power users who want to keep sources attached across start/stop cycles, or who want to manually add and remove sources for testing. In day-to-day use you can safely ignore it.

7. Accessibility Features

Spoken feedback

TeamTalk Streamer uses the Tolk library to talk to your active screen reader (NVDA, JAWS, SAPI, etc.). Announcements fire on every meaningful state change:

  • "Connecting to TeamTalk." when the TCP handshake starts.
  • "Logged in." when authentication succeeds.
  • "Joined channel." when the channel join completes.
  • "Streaming started." when audio is flowing.
  • "Disconnected." when the session ends cleanly.
  • An error message with the server's reason text if anything fails.

Audio feedback tones

Tones play alongside speech for non-verbal confirmation of common events:

  • Connect chime (rising two-note) when logged in.
  • Streaming-started fanfare (four ascending notes) when frames begin flowing.
  • Disconnect tone (falling two-note) on clean disconnect.
  • Error buzz (low two-note) on failure.

Pairing PINs and numeric codes

Any PIN or numeric code the app speaks (such as the mobile bridge pairing PIN in a future version) is spoken digit by digit — "one two three four five six" — rather than as a large number, matching how screen readers normally handle verification codes and OTPs.

8. Troubleshooting

"Connecting to TeamTalk" but never finishes

Double-check the Host and TCP port fields. If your server uses a non-default port (anything other than 10333), both TCP and UDP fields must match what the server operator configured. Firewall software on the PC can also block outgoing connections to unusual ports.

"No server is configured. Please open Server settings and enter a host first."

This is spoken when Start streaming is pressed with an empty Host field. Open Server settings, fill in the host, and save.

"Channel path '/whatever' not found on server."

Either the channel doesn't exist, or the path is misspelled. Open Server settings and use Probe server for channel list to pick the channel from the live list instead of typing the path. Channel names are case-sensitive on most TeamTalk servers.

The app is silent when I expected spoken feedback

Tolk's native DLLs are optional add-ons — the app runs without them, but then has no way to speak. Download the Tolk DLLs from the Tolk releases page and place them alongside TeamTalkStreamer.exe. A screen reader (NVDA, JAWS, etc.) must also be running.

No feedback tones

The feedback tones need an OpenAL runtime. On most systems one is already installed; if not, install OpenAL Soft. The app tolerates its absence and simply stays silent on that channel.

The app process stayed alive after I closed the window

This shouldn't happen in v1 — the shutdown path was hardened specifically against this — but if you ever see it, report the reproduction steps. The immediate workaround is to end the TeamTalkStreamer.exe process from Task Manager.

9. Where Your Settings Live

TeamTalk Streamer stores its configuration as plain JSON at:

%APPDATA%\TeamTalkStreamer\settings.json

You can open this file in any text editor to inspect or manually edit your credentials. The app uses an atomic write-then-rename pattern, so a power loss mid-save cannot corrupt the file. Passwords are stored in clear text at present; protect this file with OS-level permissions if that is a concern.

10. Roadmap

Version 1 (current)

Windows streaming of the default playback device, credentials dialog with channel browser, one-click start/stop, clean shutdown. This is the fully-shipped feature set.

Version 2 (deferred)

Companion apps for iOS and Android so your phone can stream its audio to the same TeamTalk channel through your PC over the LAN. The server-side infrastructure is already written (mDNS discovery, WebSocket + Opus protocol, PIN-based HMAC pairing, device registry) and awaiting the client apps. Version 2 is paused until a macOS toolchain is available for building the iOS companion.
